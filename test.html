The Confusion Matrix

When measuring the efficacy of any classification algorithm, we can classify the results into

one of four categories:

True positive (TP) (”Hit”): The sample is a member of the class and we’ve identified it as

such

Example: We correctly identified an anomaly

True Negative (TN) (”Correct Rejection”): The sample is not a member of the class, and

we’ve identified that it is a non-member

Example: We correctly identified a non-anomaly as “None”

False Positive (FP) (”False Alarm”, “Overestimation”) (type 1 error): The sample is not a

member of the class, but we’ve identified it as such incorrectly

Example: We incorrectly identify non-anomaly as “high” anomaly

False Negative (FN) (”Miss”, “Underestimation”) (type 2 error): The sample is a member of

the class, but we’ve failed to identify it as such

Example: We missed a “high” anomaly and labeled it as a “None”

Collectively, these outputs form a “confusion matrix”, which is pictured above. Specifically, the

confusion matrix pictured above is a binary confusion matrix, but confusion matrices can have

arbitrarily many classes.

One can think of the first term, “True” or “False”, as being indicative of the performance of the

predictive algorithm. “True” results mean that the algorithm guessed correctly, and “False”

Performance Metrics

There are four distinct metrics typically used to evaluate the efficacy of a prediction system.

These are: Accuracy, Precision, Recall, and F1.

Accuracy

Given a sample, how likely are we to make a “correct” prediction? Accuracy is defined as the

number of correct predictions over the number of total predictions, or

This can basically be thought of as the sum of the diagonol divided by the sum of the entire

confusion matrix.

Precision

What proportion of the “positive” predictions made by the system are actually positive?

Precision is defined as the number of correct positive predictions over the number of total

positive predictions, or

This value tells us how “trigger happy” our system is. If we make a LOT of positive predictions,

we may retain a high recall (and maybe even a high accuracy if the classes are imbalanced),

despite most of our positive predictions being wrong. In the context of Velix, this means we will

provide a lot of meaningless data to the customer.

Recall

What proportion of the anomalies are actually detected by the system? Recall is defined as the

number of correct positive predictions over the number of total positive samples, or

Recall can be thought of similarly to accuracy, but unlike accuracy, it only regards positive

samples. Recall is asking a question about the specific class being investigated, rather than the

performance across all classes.

Current issues and Suggested Improvements

Currently, there seem to be a few points of confusion. Namely,

1. We are actually measuring recall when we talk about measuring accuracy

2. We are not accurately measuring precision, and should have two separate precision

metrics (one for grouping and one for levels)

First of all, because it requires so little work, it would be a good idea to simply calculate ALL of

the above metrics for both grouping and levels. We can and should use a modified (to our

liking) Fβ score as the final evaluation criteria for measuring the performance of the anomaly

detection system.

Secondly, we need to fix the metrics that we are currently measuring. Our current accuracy

metric is actually the recall, as it only considers positive samples, and we should calculate the

accuracy separately. Grouping Precision should not consider any specific level class “High”,

“Med” or “Low”, and instead be a binary metric: Did we identify an anomaly or not.

Thirdly, The metrics for levels should be calculated on a per-level basis.

Overall, this would provide us with 25 total metrics (5 for grouping and 20 for levels). We

should consider the Fβ metric most closely when evaluating the system’s performance

def accuracy(tp: int, tn: int, fp: int, fn: int) -> float:

return (tp + tn) / (tp + tn + fp + fn)

def precision(tp: int, fp: int) -> float:

return tp / (tp + fp)

def recall(tp: int, fn: int) -> float:

return tp / (tp + fn)

def f1(tp: int, fp: int, fn: int) -> float:

p = precision(tp, fp)

r = recall(tp, fn)

return 2 * ((p * r) / (p + r))

def fb(tp: int, fp: int, fn: int, beta: float = 0.25) -> float:

beta *= beta

p = precision(tp, fp)

r = recall(tp, fn)

return (1 + beta) * ((p * r) / ((beta * p) + r))

def stats(tp: int, tn: int, fp: int, fn: int, beta: float = 0.25) -> dict[str, float]:

return {

"accuracy": accuracy(tp, tn, fp, fn),

"precision": precision(tp, fp),

"recall": recall(tp, fn),

"f1": f1(tp, fp, fn),

"fb": fb(tp, fp, fn, beta)

}
